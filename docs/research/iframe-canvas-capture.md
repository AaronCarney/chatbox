# Research: Iframe Content Capture to Canvas

**Date:** 2026-04-04
**Sources:** MDN Web Docs (iframe, contentDocument, drawImage, toDataURL, sandbox), html2canvas docs & GitHub issues, modern-screenshot GitHub/docs, Context7 (html2canvas, modern-screenshot), monday.com engineering blog, W3C mediacapture-screen-share issue #145, npm-compare.com
**Confidence Summary:** 8 HIGH / 3 MEDIUM / 1 LOW

## Standard Stack

**Use a two-tier capture strategy:**

1. **Canvas-based apps (Go game, DOS emulator):** Direct `drawImage()` from `iframe.contentDocument.querySelector('canvas')` onto a local canvas. Sub-millisecond, zero dependencies.
2. **DOM-based apps (Spotify track list):** Use `modern-screenshot` (`domToPng` / `domToCanvas`) on `iframe.contentDocument.body`. ~1-2s for 20-50 elements.

Do not use html2canvas. It is 3x slower, 18x larger, and has a confirmed bug with `<canvas>` element rendering (GitHub issue #1311 — canvas areas render empty).

[HIGH | sources: MDN drawImage spec, monday.com engineering benchmark, html2canvas GitHub #1311]

## Architecture Patterns

### Tier 1: Direct Canvas Capture (canvas-based apps)

```js
const iframe = document.querySelector('iframe#app');
const sourceCanvas = iframe.contentDocument.querySelector('canvas');
const targetCanvas = document.createElement('canvas');
targetCanvas.width = sourceCanvas.width;
targetCanvas.height = sourceCanvas.height;
targetCanvas.getContext('2d').drawImage(sourceCanvas, 0, 0);
// targetCanvas is now a pixel-perfect copy
```

**Why this works:**
- `drawImage()` accepts `HTMLCanvasElement` as a source (MDN spec: accepts HTMLImageElement, SVGImageElement, HTMLVideoElement, HTMLCanvasElement, ImageBitmap, OffscreenCanvas, VideoFrame).
- Same-origin iframe with `sandbox="allow-scripts allow-same-origin"` allows full `contentDocument` access (MDN: "allow-same-origin treats the iframe content as coming from the same origin").
- No tainted canvas issue because the canvas content is generated by same-origin scripts, not cross-origin images.
- Performance: `drawImage()` is a single GPU-accelerated blit. Sub-millisecond.

**Caveat for DOS emulator (js-dos):** js-dos renders to a `<canvas>` element. The canvas is same-origin. However, if js-dos loads ROM/asset files from a CDN, verify those don't taint the canvas. Since ChatBridge serves game ZIPs from `/apps/*` (same origin), this should not be an issue.

[HIGH | source: MDN drawImage spec, MDN contentDocument, MDN sandbox attribute]

### Tier 2: DOM Snapshot (DOM-based apps)

```js
import { domToPng } from 'modern-screenshot';

const iframe = document.querySelector('iframe#app');
const dataUrl = await domToPng(iframe.contentDocument.body, {
  scale: 1,
  // For cross-origin images (e.g., Spotify album art from i.scdn.co):
  // These will fail silently or be omitted unless proxied
});
```

**Why modern-screenshot over html2canvas:**
- monday.com benchmark: modern-screenshot ~7s for 10 complex widgets vs html2canvas ~21s (3x faster).
- Bundle: modern-screenshot ~185KB unpacked vs html2canvas ~3.4MB unpacked.
- modern-screenshot has better CSS support (fonts, SVGs) per monday.com evaluation.
- html2canvas has a confirmed bug rendering `<canvas>` elements (issue #1311).

[HIGH | source: monday.com engineering blog, npm-compare.com, html2canvas GitHub #1311]

### App-Type Detection

```js
function captureApp(iframe) {
  const doc = iframe.contentDocument;
  const canvas = doc.querySelector('canvas');

  if (canvas && canvas.width > 0 && canvas.height > 0) {
    // Canvas-based app: direct capture
    const target = document.createElement('canvas');
    target.width = canvas.width;
    target.height = canvas.height;
    target.getContext('2d').drawImage(canvas, 0, 0);
    return target.toDataURL('image/png');
  }

  // DOM-based app: use modern-screenshot
  return domToPng(doc.body, { scale: 1 });
}
```

[HIGH | source: architectural pattern derived from verified primitives above]

## Don't Hand-Roll

1. **DOM-to-image rendering.** Do not attempt to manually serialize DOM to SVG foreignObject and render to canvas. This is what modern-screenshot does internally, and it handles edge cases (font embedding, CSS computed styles, SVG serialization, character escaping) that take thousands of lines.

2. **Cross-origin image proxying.** modern-screenshot and html2canvas both support `useCORS: true` but this requires the image server to send CORS headers. Spotify's `i.scdn.co` CDN does send CORS headers for album art. If it didn't, you'd need a server-side proxy. Do not write a client-side CORS workaround.

3. **Canvas taint detection.** monday.com built a utility to detect tainted canvases before export. For ChatBridge, this is unnecessary because all apps are same-origin (`/apps/*`). The only cross-origin content is Spotify album art images within DOM, not drawn onto a canvas element.

[HIGH | source: monday.com engineering blog, html2canvas docs]

## Common Pitfalls

1. **`sandbox="allow-scripts allow-same-origin"` security warning.** MDN explicitly warns: "When the embedded document has the same origin as the embedding page, it is strongly discouraged to use both allow-scripts and allow-same-origin, as that lets the embedded document remove the sandbox attribute." The sandboxed iframe can run `document.querySelector('iframe').removeAttribute('sandbox')` from within, effectively escaping the sandbox. **Mitigation:** For ChatBridge, the iframe content is first-party (admin-curated apps served from `/apps/*`), not untrusted third-party code. The risk is that a compromised app could escape sandboxing. Accept this tradeoff for the capture feature, or use CSP `frame-src` as an additional layer.

[HIGH | source: MDN iframe sandbox documentation]

2. **html2canvas renders canvas elements as empty.** Confirmed bug (GitHub #1311, labeled "Bug", closed as completed). Root cause: html2canvas removes the container from the DOM before rendering, preventing canvas content capture. This affects ECharts, Chart.js, and any canvas-based rendering. This is why the direct `drawImage()` approach is mandatory for canvas-based apps.

[HIGH | source: html2canvas GitHub issue #1311]

3. **Cross-origin images taint the canvas.** If `domToPng` renders a DOM containing `<img src="https://i.scdn.co/...">`, and that image doesn't have CORS headers, the resulting canvas is tainted and `toDataURL()` throws `SecurityError`. Spotify's CDN does serve images with CORS headers (`Access-Control-Allow-Origin: *`), but verify at runtime. Fallback: proxy images through ChatBridge's backend.

[MEDIUM | source: MDN toDataURL SecurityError docs, html2canvas CORS docs; Spotify CDN CORS claim from web search, not verified against official Spotify docs]

4. **Performance: html2canvas blocks the main thread.** html2canvas processes DOM synchronously on the main thread. For 883 DOM nodes: ~8 seconds. For 2660 nodes: ~66 seconds. It does not yield to the event loop during rendering. modern-screenshot is also main-thread but ~3x faster due to SVG-based approach rather than pixel-by-pixel canvas painting.

[MEDIUM | source: html2canvas GitHub issues #1250, #987, #263; monday.com benchmark]

5. **`iframe.contentDocument` returns `null` for cross-origin.** If any iframe loads content from a different origin (even with `allow-same-origin` in sandbox), `contentDocument` returns `null`. All ChatBridge apps must be served from the same origin for capture to work.

[HIGH | source: MDN HTMLIFrameElement.contentDocument]

## Code Examples

### Direct canvas capture from same-origin iframe

```js
// Assumes: iframe serves content from same origin with sandbox="allow-scripts allow-same-origin"
const iframe = document.getElementById('game-iframe');
const gameCanvas = iframe.contentDocument.querySelector('canvas');

// Create capture canvas
const capture = document.createElement('canvas');
capture.width = gameCanvas.width;
capture.height = gameCanvas.height;
const ctx = capture.getContext('2d');
ctx.drawImage(gameCanvas, 0, 0);

// Export as data URL or blob
const dataUrl = capture.toDataURL('image/png');
// or
capture.toBlob((blob) => {
  // Send blob to chat as inline image
}, 'image/png');
```

[HIGH | source: MDN drawImage accepts HTMLCanvasElement, MDN toDataURL, MDN toBlob]

### modern-screenshot DOM capture

```js
import { domToPng, domToCanvas } from 'modern-screenshot';

const iframe = document.getElementById('spotify-iframe');
const body = iframe.contentDocument.body;

// As data URL (PNG)
const png = await domToPng(body, {
  scale: 1,          // 1x resolution sufficient for chat preview
  backgroundColor: '#ffffff',
});

// As canvas (if further manipulation needed)
const canvas = await domToCanvas(body, { scale: 1 });
```

[HIGH | source: modern-screenshot README, Context7 modern-screenshot docs]

### Hybrid capture with type detection

```js
import { domToPng } from 'modern-screenshot';

async function captureIframeContent(iframe) {
  const doc = iframe.contentDocument;
  if (!doc) throw new Error('Cannot access iframe document (cross-origin?)');

  // Check for canvas-based app
  const canvas = doc.querySelector('canvas');
  if (canvas && canvas.width > 0 && canvas.height > 0) {
    const buf = document.createElement('canvas');
    buf.width = canvas.width;
    buf.height = canvas.height;
    buf.getContext('2d').drawImage(canvas, 0, 0);
    return buf.toDataURL('image/png');
  }

  // Fallback: DOM-based app
  return domToPng(doc.body, { scale: 1 });
}
```

[HIGH | source: composed from verified MDN APIs + modern-screenshot API]

## Validation Architecture

**Unit tests:**
- Mock `iframe.contentDocument` with a JSDOM-like structure containing a `<canvas>` element. Verify `drawImage` path is taken.
- Mock `iframe.contentDocument` with DOM-only content (no canvas). Verify `domToPng` path is taken.
- Test null `contentDocument` (cross-origin simulation) throws descriptive error.

**Integration tests:**
- Serve a test HTML page at `/apps/test-canvas` with a `<canvas>` that draws a known pattern (e.g., red rectangle). Capture and verify pixel data matches.
- Serve a test HTML page at `/apps/test-dom` with styled DOM elements. Capture and verify image is non-empty and dimensions are correct.

**Manual verification:**
- Capture the Go game board mid-play. Verify the stones are visible.
- Capture the DOS emulator during gameplay. Verify the frame is not blank.
- Capture the Spotify track list. Verify album art images are present (not blank/broken).

[HIGH | source: standard testing patterns for canvas/DOM APIs]

## Unverified Findings

1. **Spotify CDN CORS headers.** Claim: `i.scdn.co` serves images with `Access-Control-Allow-Origin: *`. This is widely reported in developer forums but not confirmed against official Spotify documentation. If false, album art will be omitted from DOM captures or cause tainted canvas errors. Mitigation: test at runtime, fall back to server-side proxy.
[LOW -- unverified: no official Spotify CDN documentation found confirming CORS headers on image assets]

2. **modern-screenshot handles `<canvas>` elements in DOM.** The monday.com article shows they had to build tainted-canvas detection/removal before using modern-screenshot, implying modern-screenshot does attempt to capture canvas elements but fails on tainted ones. For same-origin untainted canvases, it likely works, but no explicit documentation confirms this. The hybrid approach (direct drawImage for canvas apps) avoids this uncertainty entirely.
[LOW -- unverified: modern-screenshot README and docs do not explicitly document canvas element handling]

3. **No native DOM-to-image browser API exists.** W3C mediacapture-screen-share issue #145 discussed adding `createImageBitmap()` for DOM nodes but closed without implementation. No `Element.captureScreenshot()`, `element.toBlob()`, or similar API exists in any browser. The only native approach is manual canvas rendering or the library-based approaches above.
[MEDIUM | source: W3C GitHub issue #145 (closed, no implementation); MDN has no such API documented]
