8-Layer Defense Architecture

Every interaction passes through eight independent safety layers. No single point of failure can compromise student safety. Each layer operates independently — even if one is bypassed, the remaining seven continue to protect.

Student Request (HTTPS) | v [L1] Transport Security ──── Security headers (CSP, HSTS, X-Frame-Options), | CORS whitelist, OAuth CSRF state tokens v [L2] Rate Limiting ────────── IP-based throttling, chat-specific limits | v [L3] Authentication ───────── Clerk session validation, JWT verification | v [L4] PII Stripping ────────── Regex removal of emails, phones, SSNs | from all message roles before processing v [L5] Prompt Injection ─────── Delimiter-wrapped tool results, system prompt | hardening, leak detection (fragment matching) v [L6] LLM + Tool Calls ─────── GPT-4o strict JSON Schema tool params, | tool call limits (10/turn, 5-deep chain) v [L7] Output Validation ────── Schema validation, size caps (2KB per result), | OpenAI Moderation API on text outputs v [L8] Client-Side Safety ───── On-device NSFWJS model, iframe sandboxing, real-time content monitoring, safety overlay

Defense Layers in Detail

🌐
L1: Transport Security
Security headers (CSP, HSTS, X-Frame-Options) via Helmet. CORS whitelist rejects requests from unauthorized origins. Spotify OAuth uses random state tokens validated server-side to prevent cross-site request forgery.
Why: Prevents man-in-the-middle attacks, clickjacking, and cross-origin exploits before any application code runs.
📊
L2: Rate Limiting
IP-based throttling on all API endpoints (100 req/15 min). Stricter limits on chat (20 req/min) and moderation endpoints. Prevents abuse and controls API costs.
Why: Stops automated attacks and prevents a single user from overwhelming the system or running up API bills.
🔑
L3: Authentication
Clerk-based session management with JWT verification on all chat endpoints. Pseudonymized identifiers (HMAC-SHA256, daily-rotating) in logs — real user IDs never stored or logged.
Why: Ensures only authorized users access the platform. Pseudonyms provide audit trails without exposing student identity.
🧹
L4: PII Stripping
Regex-based scrubbing of emails, phone numbers, SSN patterns, and street addresses from all message content — user, assistant, and tool messages alike — before any LLM processing.
Why: Even if a student accidentally types personal information, it's removed before the AI sees it or it reaches any logs.
🛡️
L5: Prompt Injection Defense
Tool results wrapped in unique salt-randomized delimiters with source attribution. System prompt explicitly forbids following instructions from tool results. Leak detection scans outputs for system prompt fragments.
Why: Prevents malicious content in third-party data from hijacking the AI. The #1 attack vector for LLM applications (OWASP LLM01).
🤖
L6: LLM + Tool Calls
GPT-4o with OpenAI strict mode — all tool parameters validated against JSON Schema. Max 10 tool calls per turn, 5-deep chaining limit. 8K token input budget with progressive history trimming. Filler text before tool calls suppressed.
Why: Constrains the AI to well-defined actions only. Prevents runaway execution and context window exploitation.
📋
L7: Output Validation
AJV schema validation on all tool results. Hard 2KB size cap per result. OpenAI Moderation API checks text content for violence, self-harm, sexual content, and hate speech. Unknown tools rejected outright.
Why: Validates everything before the student sees it. Size caps prevent data exfiltration; moderation catches harmful AI-generated content.
👁️
L8: Client-Side Safety
On-device NSFWJS model (TensorFlow.js Web Worker) monitors iframe content every 5 seconds via canvas snapshots. Hysteresis state machine prevents false-positive flickering. Hard blocks for sexual/minors and self-harm content are permanent and cannot be dismissed.
Why: Last line of defense. Even if an embedded app displays harmful content, the student is protected before any server round-trip.

Iframe Sandboxing

All five embedded apps (Chess, Go, Nature Explorer, DOS Arcade, Spotify) run inside sandboxed iframes. Think of each iframe as a locked room — the app can run code, but it cannot reach anything outside its walls.

  • Sandbox policy: allow-scripts only — apps cannot access parent DOM, read cookies, navigate away, open popups, or submit forms
  • Credentialless: The credentialless attribute prevents iframes from accessing any stored browser credentials (cookies, HTTP auth, sessions)
  • Feature policy: allow="" denies access to camera, microphone, geolocation, and all other sensitive browser APIs
  • Referrer policy: no-referrer prevents URL information from leaking to embedded content
  • PostMessage protocol: All parent-child communication uses a versioned schema (CHATBRIDGE_V1) with structured envelopes. Messages without the correct schema are silently dropped. MessageChannel ports isolate request/response flows.
  • Cross-app isolation: Apps cannot read or write other apps' state. Save data validated as plain objects, capped at 512KB per app per session. Source validation prevents cross-app spoofing.
  • No external navigation: Students cannot be redirected to external sites from within any app (COPPA compliance)

OAuth Security (Spotify)

Spotify integration uses a server-side OAuth proxy — the student never handles tokens directly.

  • CSRF protection: Each OAuth flow generates a cryptographically random state token (128-bit). The callback validates this token server-side and rejects unrecognized values. Tokens are single-use — deleted after exchange.
  • Server-side token exchange: The Spotify client secret never leaves the server. Token exchange happens server-side; the client only sees an opaque session reference.
  • Session-scoped: Spotify tokens are tied to ephemeral session IDs, not persistent user accounts. No long-lived refresh tokens stored.
  • Error sanitization: Error messages are HTML-escaped before rendering to prevent XSS via OAuth error callbacks.

Content Filtering

Age-appropriate content is enforced at multiple levels across all apps.

  • Nature Explorer: Multi-layer filtering — taxonomic blocklist (removes parasites, disturbing species), content name filter, taxon type allowlist (only Animalia, Plantae, Fungi), license-filtered images (Creative Commons only), Wikipedia HTML stripped via textContent (no innerHTML anywhere in client apps)
  • DOS Arcade: Curated 17-game catalog reviewed for age-appropriateness. M-rated titles excluded. No user-uploaded content.
  • Spotify: Search returns public catalog only. Session-scoped auth prevents access to personal libraries.
  • DOM safety: All client-side apps use document.createElement + textContent for rendering. No innerHTML is used anywhere, preventing cross-site scripting (XSS) via injected markup.

Data Privacy

  • Ephemeral sessions: Chat history exists only in browser memory. Closing the tab erases the entire conversation — nothing is saved on any server.
  • No student data storage: No names, emails, grades, or demographic information is collected or stored anywhere in the system.
  • Pseudonymized audit trail: Server logs use HMAC-SHA256 pseudonyms that rotate daily. Even the developers cannot reverse these back to real user identifiers. Traces are recorded via Langfuse for debugging without exposing PII.
  • No external tracking: No analytics, ad trackers, or third-party data collection scripts.
  • App state isolation: Each iframe app's save data is scoped to the current session and sandboxed from other apps.

OWASP LLM Top 10 Coverage

LLM01: Prompt Injection
Delimiter wrapping with random salts, system prompt hardening, strict tool schemas, leak detection, filler suppression
LLM02: Sensitive Info Disclosure
PII stripping, pseudonymized logs, ephemeral sessions, no student data stored
LLM03: Supply Chain
Sandboxed iframes, credentialless attribute, origin validation, CORS whitelist
LLM05: Improper Output
Schema validation, HTML stripping, size caps, OpenAI Moderation API
LLM06: Excessive Agency
Tool call limits (10/turn), chaining depth cap (5), least-privilege sandbox, no external navigation
LLM07: System Prompt Leakage
Anti-leak instructions, fragment-based leak detection, filler suppression, no credentials in prompt
LLM10: Unbounded Consumption
Rate limiting, token budget (8K), max_tokens cap (1024), request timeouts (8s on external APIs)
